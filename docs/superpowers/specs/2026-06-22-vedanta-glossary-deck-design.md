# Vedanta Glossary Word Deck ‚Äî Design

**Date:** 2026-06-22
**Status:** Approved (ready for implementation planning)

## Goal

A new Anki deck of **single Vedanta terms** (a word or two), complementing the
verse deck. Front: the Sanskrit term in Devanagari. Back: IAST + English meaning.
No audio.

## Source

The **arshabodha "Vedanta-Sanskrit Glossary"** PDF
(`https://arshabodha.org/wp-content/uploads/abc/teachings/Vedanta-Sanskrit-Glossary.pdf`)
‚Äî compiled by John Warne from Swami Dayananda Saraswati's three-year Vedanta &
Sanskrit course. Authoritative, Vedanta-specific, and from arshabodha (the user's
preferred org). **~2,394 entries.**

### What the PDF gives us (verified)

The PDF has **no usable Unicode text layer** ‚Äî it renders with subsetted,
custom-encoded TrueType fonts that have no ToUnicode map, so its Devanagari is
unrecoverable as text. But the layout is extremely regular: each entry is one row
with three columns identified by font:

- **`TT2CEt00`** (x‚âà90, left margin) ‚Äî IAST headword, mangled by a fixed font
  substitution (`ƒÅ`‚Üí`√§`, `ƒ´`‚Üí`√©`, `≈´`‚Üí`√º`, `·∏•`‚Üí`√π`, `·π£`‚Üí`√±`, ‚Ä¶).
- **`TT2CDt00`** ‚Äî Devanagari (garbage glyphs ‚Äî discarded).
- **`TT2D1t00`** ‚Äî English definition (extracts cleanly).
- **`TT2D0t00`** ‚Äî italic: grammatical abbreviations (`comp.`, `ind.`, `nom.
  sing.`, `p.p.p.`) and IAST cross-reference terms inside definitions (also
  mangled by the same kind of substitution).

So from the PDF we can reliably recover, per entry, the **IAST headword** and the
**English definition** (the latter including embedded IAST terms once decoded).
The **Devanagari is regenerated by transliteration**, not read from the PDF.

## Pipeline

**Source of truth.** `data/vedanta.json` is the committed source of truth. The
PDF and its **extractor are one-time, disposable bootstrap scaffolding** ‚Äî they
don't need to be robust or handle every edge case, aren't part of the runtime,
and can be deleted once the JSON is built and QA-clean. We never need to re-parse
the PDF. Any residual error ‚Äî the 2 multi-word entries, a stray decode slip ‚Äî is
**fixed directly in `data/vedanta.json`**, not by hardening the extractor (only
fix the tool when an error is *systematic* and cheaper to fix once than to
hand-edit many rows).

The **IAST‚ÜîDevanagari transliterator is the exception: it is kept** as a
committed, tested library (`lib/iast_devanagari.rb`) ‚Äî it's a genuinely reusable
tool (and unblocks a future per-word verse deck), used here to generate the
Devanagari column and to power the round-trip QA check. The rest of the runtime
is a loader + a pure-transform generator over the committed JSON.

### Committed library ‚Äî kept

1. **`lib/iast_devanagari.rb`** (pure Ruby, tested): deterministic IAST ‚Üí
   Devanagari transliteration ‚Äî independent vowels, consonants with inherent-`a`
   and virama clustering, mƒÅtrƒÅs, anusvƒÅra (`·πÉ`‚Üí ý§Ç), visarga (`·∏•`‚Üí ý§É), vocalic
   `·πõ ·πù ·∏∑ ·∏π`, diphthongs `ai/au`, etc. Also provides the **reverse**
   (`to_iast(devanagari)`) used by the round-trip QA check (method B);
   `to_iast(to_devanagari(x)) == x` for valid IAST. Reusable for a future
   per-word verse deck.

### Bootstrap ‚Äî one-time, disposable (`bootstrap/`)

2. **`bootstrap/extract_vedanta.py`** (Python + `pypdf`): per-chunk
   font+position visitor ‚Üí for each entry row, `{iast, definition}`:
   - **IAST headword** = first `TT2CEt00` chunk, decoded via the `TT2CEt00`
     WinAnsi‚ÜíIAST map.
   - **definition** = `TT2D1t00` text (+ `TT2D0t00` italics), embedded IAST
     decoded via the `TT2D0t00` map, abbreviations expanded; optionally wrap
     italic spans in `<i>‚Ä¶</i>`.
   - First-headword/first-Devanagari rule (extra Devanagari simply dropped); the
     page-1 header (title, intro, "Abbreviations:" legend) is skipped because the
     entry-row rule (leftmost chunk is a `TT2CEt00` headword at x‚âà90) excludes it.
   - Keep it simple ‚Äî no override map for the 2 multi-word entries; those are
     fixed in the JSON afterward.
   - `pypdf` installs cleanly (`pip install pypdf`); pdfplumber/pdfminer.six do
     **not** build on this machine's free-threaded CPython 3.13 (their
     `cryptography`/CFFI dependency is unsupported there).
3. **`bootstrap/build_vedanta.rb`** (one-time): runs the extractor, calls
   `lib/iast_devanagari.rb` to add the `devanagari` column, and writes the first
   cut of `data/vedanta.json` = `[{iast, devanagari, definition}]`. Disposable
   once the QA-clean JSON is committed.

### Runtime ‚Äî committed, kept

4. **`data/vedanta.json`** ‚Äî the committed source of truth.
5. **`lib/vedanta.rb`** ‚Äî loader (`Vedanta.load`), mirroring `lib/gita.rb`.
6. **`lib/generators/vedanta.rb`** (`--vedanta`) ‚Äî pure transform over
   `data/vedanta.json`.

## Multi-word entries

Only **2** of ~2,394 entries carry more than one Devanagari before the
definition:

- **`a` / `an`** (ý§Ö / ý§Öý§®ý•ç) ‚Äî the prefix, two headword forms sharing one
  definition.
- **`dyo`** (ý§¶ý•çý§Øý•ã) ‚Äî one headword plus an inline *"Nom. sing."* form (ý§¶ý•çý§Øý•åý§É).

The parser uses one uniform rule ‚Äî **first IAST headword + first Devanagari =
the card; all remaining text ‚Üí definition** ‚Äî which is correct for every other
entry and degrades gracefully on these two (the extra Devanagari, being in the
undecodable font, is simply dropped). Since `data/vedanta.json` is the source of
truth, the two corrections (note the `an` form on the `a` entry; keep `dyo`'s
stem) are simply **edited directly into `data/vedanta.json`** during QA ‚Äî no
special-casing in the throwaway parser.

## Abbreviations legend

The PDF's top-of-page-1 legend defines the grammatical abbreviations used inside
definitions: `comp.` = compound, `ind.` = indeclinable, `lit.` = literally,
`nom. sing.` = nominative singular, `p.p.p.` = past passive participle. Because an
Anki card can't refer back to a legend, the extractor carries this fixed map and
**expands these abbreviations inline** in each definition (`ind.` ‚Üí
`indeclinable`, etc.) so every card is self-contained. The map lives as a small
constant in `bootstrap/extract_vedanta.py`; if a new abbreviation surfaces, it's
added there (or the affected definitions are fixed in the JSON).

## Deck & cards

- New deck **`üïâÔ∏è Vedanta Glossary`** (constant `Anki::VEDANTA_DECK`). No audio.
- **Front:** the Devanagari term, centered/large (`Anki.glyph_front`-style, but a
  word may be several glyphs ‚Äî a single `<big>` wrap, like the verse front).
- **Back:** IAST (bold) + English definition.
- **Key:** `vedanta:<iast>` (stable GUID, so re-import updates).
- `requires_letters?` ‚Üí false; `audio_files` ‚Üí `[]` (no media-copy prompt).
- ~2,394 cards.

## QA of `data/vedanta.json` (gating ‚Äî must pass before the deck is built)

This is the critical step: the Devanagari is machine-transliterated and the IAST
/definitions are font-decoded, so `data/vedanta.json` must be proven correct
**before** `lib/generators/vedanta.rb` is run. QA uses four independent methods,
layered so cheap automated checks catch most errors and the expensive visual pass
confirms the rest. The deck generator is implemented/run only after QA is clean.

**A. Structural validation** (`bootstrap/qa_vedanta.rb`, automated, must be 0 failures):
- **Coverage:** entry count equals the PDF's (‚âà2,394), and per-page entry-row
  counts match the extractor's per-page output (no rows silently dropped/merged).
- **Non-empty:** every record has non-empty `iast`, `devanagari`, `definition`.
- **IAST charset:** `iast` contains only valid IAST (`[a-zƒÅƒ´≈´·πõ·πù·∏∑·∏π·πÖ√±·π≠·∏ç·πá≈õ·π£·πÉ·∏•]`,
  space, hyphen). **Any residual legacy-font character** (`√§ √© √Ø √∂ √º √± √ß √π √Ý √® √¨
  √≤ √≥ √™ √≠ ‚Ä¶`) anywhere in `iast` **or** `definition` means an unmapped glyph ‚Üí
  fail (this is what catches an incomplete font-decode map).
- **Devanagari charset:** `devanagari` contains only U+0900‚ÄìU+097F (+ space);
  no Latin/leftover glyph-soup.
- **Definitions:** no Devanagari-glyph-soup leaked from the wrong column; no bare
  unexpanded abbreviations (`comp.`/`ind.`/`lit.`/`nom. sing.`/`p.p.p.`).
- **Keys:** report duplicate `vedanta:<iast>` keys.

**B. Round-trip transliteration** (automated, deterministic ‚Äî high power): reverse
the transliterator (Devanagari ‚Üí IAST) for every entry and assert it equals the
source `iast` (normalized). This pinpoints transliterator bugs across all 2,394
entries without any vision, and every mismatch is a concrete bug to fix.

**C. Full visual cross-reference against the PDF** (the core deliverable): render
all 59 PDF pages and, for **every** entry, compare both the transliterated
**Devanagari** and the decoded **IAST** to the PDF's printed forms, plus
spot-check the definition text. Because this is ~2,394 entries over 59 pages, it
is parallelized (a batch of read-only subagents, each assigned a page range, each
given the corresponding `vedanta.json` slice and returning a structured list of
mismatches). Output: a single **discrepancy report** (entry, field, PDF form,
json form, note).

**D. Iterate to clean:** fix every issue A‚ÄìC surface. Prefer **editing
`data/vedanta.json` directly** (it's the source of truth) ‚Äî including the 2
multi-word entries and any one-off slips. Only re-run the bootstrap (and fix the
transliterator or font-decode map) when an error is **systematic** ‚Äî affecting
many entries, where one fix is cheaper than many hand-edits ‚Äî then re-apply the
hand-fixes. Re-run A‚ÄìC until there are zero failures and zero unexplained visual
discrepancies. **Only then** is the deck generated. After it's clean and
committed, the `bootstrap/` scripts may be removed.
