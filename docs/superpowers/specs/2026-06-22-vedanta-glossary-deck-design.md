# Vedanta Glossary Word Deck — Design

**Date:** 2026-06-22
**Status:** Approved (ready for implementation planning)

## Goal

A new Anki deck of **single Vedanta terms** (a word or two), complementing the
verse deck. Front: the Sanskrit term in Devanagari. Back: IAST + English meaning.
No audio.

## Source

The **arshabodha "Vedanta-Sanskrit Glossary"** PDF
(`https://arshabodha.org/wp-content/uploads/abc/teachings/Vedanta-Sanskrit-Glossary.pdf`)
— compiled by John Warne from Swami Dayananda Saraswati's three-year Vedanta &
Sanskrit course. Authoritative, Vedanta-specific, and from arshabodha (the user's
preferred org). **~2,394 entries.**

### What the PDF gives us (verified)

The PDF has **no usable Unicode text layer** — it renders with subsetted,
custom-encoded TrueType fonts that have no ToUnicode map, so its Devanagari is
unrecoverable as text. But the layout is extremely regular: each entry is one row
with three columns identified by font:

- **`TT2CEt00`** (x≈90, left margin) — IAST headword, mangled by a fixed font
  substitution (`ā`→`ä`, `ī`→`é`, `ū`→`ü`, `ḥ`→`ù`, `ṣ`→`ñ`, …).
- **`TT2CDt00`** — Devanagari (garbage glyphs — discarded).
- **`TT2D1t00`** — English definition (extracts cleanly).
- **`TT2D0t00`** — italic: grammatical abbreviations (`comp.`, `ind.`, `nom.
  sing.`, `p.p.p.`) and IAST cross-reference terms inside definitions (also
  mangled by the same kind of substitution).

So from the PDF we can reliably recover, per entry, the **IAST headword** and the
**English definition** (the latter including embedded IAST terms once decoded).
The **Devanagari is regenerated by transliteration**, not read from the PDF.

## Pipeline

Mirrors the project's pattern: a one-time data-prep step produces a committed
`data/*.json`; the generator is a pure transform.

1. **`tools/extract_vedanta.py`** (Python + `pypdf`, one-time dev tooling):
   parses the PDF with a per-chunk font+position visitor, groups chunks into rows,
   and for each entry row emits `{iast, definition}`:
   - **IAST headword** = first `TT2CEt00` chunk, decoded via the `TT2CEt00`
     WinAnsi→IAST map.
   - **definition** = the `TT2D1t00` text (+ `TT2D0t00` italics), with embedded
     IAST decoded via the `TT2D0t00` map; optionally wrap italic spans in
     `<i>…</i>`.
   - Additional Devanagari chunks (the 2 multi-word cases) are ignored by the
     first-headword/first-term rule (see below).
   - The **page-1 header is skipped** — the title, the intro paragraph, and the
     "Abbreviations:" legend are not entries. The entry-row rule (leftmost chunk
     is a `TT2CEt00` headword at x≈90) already excludes them (the intro is
     `Times-Roman`; the legend lines start with an italic `TT2D0t00`
     abbreviation, not a left-margin headword); the extractor asserts the first
     parsed entry is `a` (अ) as a guard.
   - Output: `data/vedanta_raw.json` = `[{iast, definition}]`.
   - `pypdf` is pure-Python and installs cleanly (`pip install pypdf`); pdfplumber
     /pdfminer.six do **not** build on this machine's free-threaded CPython 3.13
     (their `cryptography`/CFFI dependency is unsupported there), which is why we
     use `pypdf` directly.
2. **`lib/iast_devanagari.rb`** (pure Ruby, tested): deterministic IAST →
   Devanagari transliteration — independent vowels, consonants with inherent-`a`
   and virama clustering, mātrās, anusvāra (`ṃ`→ ं), visarga (`ḥ`→ ः), vocalic
   `ṛ ṝ ḷ ḹ`, diphthongs `ai/au`, etc. Reusable (also unblocks a future
   per-word verse deck).
3. **`fetch_vedanta.rb`** (Ruby, the standalone entry point): ensures the PDF is
   present, runs `tools/extract_vedanta.py`, transliterates each `iast` →
   Devanagari, and writes committed **`data/vedanta.json`** =
   `[{iast, devanagari, definition}]`. Validates entry count and that no field is
   empty.
4. **`lib/generators/vedanta.rb`** (`--vedanta`): pure transform over
   `data/vedanta.json` (read via `Gita`-style loader `lib/vedanta.rb`).

## Multi-word entries

Only **2** of ~2,394 entries carry more than one Devanagari before the
definition:

- **`a` / `an`** (अ / अन्) — the prefix, two headword forms sharing one
  definition.
- **`dyo`** (द्यो) — one headword plus an inline *"Nom. sing."* form (द्यौः).

The parser uses one uniform rule — **first IAST headword + first Devanagari =
the card; all remaining text → definition** — which is correct for every other
entry and degrades gracefully on these two (the extra Devanagari, being in the
undecodable font, is simply dropped). The two corrections (note the `an` form on
the `a` entry; keep `dyo`'s stem) live as a small **override map keyed by IAST
headword** in `tools/extract_vedanta.py`, so they survive re-extraction — no
hand-editing of generated files, and no special-casing in the parsing loop.

## Abbreviations legend

The PDF's top-of-page-1 legend defines the grammatical abbreviations used inside
definitions: `comp.` = compound, `ind.` = indeclinable, `lit.` = literally,
`nom. sing.` = nominative singular, `p.p.p.` = past passive participle. Because an
Anki card can't refer back to a legend, the extractor carries this fixed map and
**expands these abbreviations inline** in each definition (`ind.` →
`indeclinable`, etc.) so every card is self-contained. The map lives as a small
constant in `tools/extract_vedanta.py`; if a new abbreviation surfaces, it's added
there.

## Deck & cards

- New deck **`🕉️ Vedanta Glossary`** (constant `Anki::VEDANTA_DECK`). No audio.
- **Front:** the Devanagari term, centered/large (`Anki.glyph_front`-style, but a
  word may be several glyphs — a single `<big>` wrap, like the verse front).
- **Back:** IAST (bold) + English definition.
- **Key:** `vedanta:<iast>` (stable GUID, so re-import updates).
- `requires_letters?` → false; `audio_files` → `[]` (no media-copy prompt).
- ~2,394 cards.

## QA cross-reference (required deliverable)

After generation, render the PDF pages (vision) and compare **both** the
transliterated **Devanagari** and the decoded **IAST** of each entry against the
PDF's printed forms. Produce a discrepancy report listing every term whose
generated Devanagari or IAST looks different from the source. Fix the bugs those
reveal (transliterator rules or font-decode map), regenerate, and re-check until
the report is clean (or remaining diffs are explained, e.g. the 2 multi-word
entries).

## Shared-code changes

- `lib/anki.rb`: add `VEDANTA_DECK = "🕉️ Vedanta Glossary"`.
- `main.rb`: register `Generators::Vedanta` (the `--vedanta` flag, `--all`,
  `--list`, and lazy-letters loading all follow automatically).
- `data/vedanta.json` committed like other `data/*.json`; the source PDF is
  downloaded by `fetch_vedanta.rb` (not committed).

## Testing

- `lib/iast_devanagari.rb`: unit tests for vowels, consonant+vowel (mātrā),
  consonant clusters (virama), anusvāra, visarga, vocalic `ṛ`, diphthongs, and a
  few real glossary headwords (e.g. `abhāvaḥ`→अभावः, `mokṣaḥ`→मोक्षः,
  `jñānam`→ज्ञानम्).
- `lib/generators/vedanta.rb`: card front/back from a fixture entry — key
  `vedanta:<iast>`, Devanagari on front, IAST + definition on back, no audio.
- `fetch_vedanta.rb`: integration — run it; assert ~2,394 records, all fields
  non-empty.
- The QA cross-reference pass above.

## Out of scope

- Per-word *verse* deck (separate future effort; would reuse
  `lib/iast_devanagari.rb`).
- Audio.
- Any change to the alphabet or Gita decks.
